**วันที่:** 2026-07-12  
**Repo:** `sangkan-clean`  
**สถานะ:** ร่างรออนุมัติ — ปรับมาตรฐาน UX เป็นเป้า **10/10/10** แล้ว (§11)  
**มติ:** แนวทาง A (เก็บหน้าจอเดิม จัดกระดูกข้อมูลใหม่) + แกนหลักเป็น TypeScript + จัดใหญ่ให้จบก่อนเปิดออนไลน์ (ตัวเลือก 2) + UX 10/10

## 1. ปัญหา

`sangkan-clean` มีหน้าจอครอบคลุมอยู่แล้ว (CRM, ใบเสนอราคา, ปฏิบัติการ, HR, คลัง, การเงิน) แต่ข้อมูลยังเป็นแบบชีตหลวมๆ เชื่อมกันไม่แน่น:

- อ้างอิงด้วยชื่อข้อความมากกว่า ID คงที่
- สถานะขั้น CRM กระจายอยู่ทั้ง `AppSetting` และหลายชีต
- มีทางเดินซ้ำ (PR/ใบเบิก, PO/จัดซื้อ, Inventory/Itemcatalog)
- ไม่มีกระดูกสันหลังเดียวแบบ Deal → Job → Document → Payment
- ใช้ JavaScript ทั้งก้อน ทำให้หลุดลิงก์จำเป็นได้ง่าย (`customerId`, `dealId`)

ระบบยัง**ไม่เปิดใช้งานจริง** ทีมจึงเลือกจัดแกนให้คล้องตัว**ก่อน**ขึ้นออนไลน์ แทนการแพตช์ทีละจุดหลังเปิดแล้ว

ระบบที่เกี่ยวข้อง (`cleaning-seo-website` ส่วน Growth + LINE Ops) **อยู่นอกขอบเขต**รอบนี้ ยกเว้นฟิลด์อ่อน (`lineUserId`) และการแจ้ง LINE ที่มีอยู่แล้ว

## 2. เป้าหมาย

1. **กระดูกเส้นเดียว:** Customer → Deal → (Survey / Quotation / Job / Document / Payment / Expense)
2. **แม่นยำ:** เอกสารและงานต้องอ้าง `customerId` / `dealId` เสมอ (และงาน assign ด้วย `employeeId`)
3. **รวดเร็ว:** happy path ไม่ต้องคีย์ชื่อลูกค้า/พนักงานซ้ำข้ามจอ
4. **TypeScript:** แกนใหม่ (schema, API, domain services) เป็น TS พร้อมชนิดเข้มสำหรับ stage และชนิดเอกสาร
5. **พร้อมเปิดใช้:** แต่ละ workflow มีทางหลักทางเดียว ซ่อนหรือตัดจอซ้ำ/ยังไม่เสร็จ
6. **หน้า Topology Graph:** แผนที่ทั้ง**ฟังก์ชันภายใน ERP** และจุดเชื่อมภายนอก (LINE / เว็บ) — ให้ทีมเห็นว่าระบบทำอะไรเชื่อมอะไรได้ในที่เดียว

### สิ่งที่ไม่ทำในรอบนี้

- แปลงหน้าใหญ่ทุกไฟล์เป็น TS ก่อนเปิด (ย้ายทีละโมดูลหลังแกนนิ่ง)
- รวมสมุดบิล LINE Sheets กับ ERP เป็นบัญชีเล่มเดียว
- ดึงลีดจากเว็บอัตโนมัติ, บอท SEO, บอทโซเชียล
- ย้ายไปใช้ Peak/FlowAccount แทน
- ระบบลงเวลาเข้างาน HR (ยังเป็นของอนาคต)
- สถานะบอท GitHub สดบน Topology (ทีหลังได้ — รอบแรกใช้แผนที่ static/จาก registry ฟังก์ชัน)

## 4. โมเดลโดเมน

### 4.1 Master (แหล่งความจริงของคน)

**Customer (ลูกค้า)**

- `id`, `name`, `taxId?`, `branchCode?`, `address?`, `zip?`, `phone?`, `email?`
- `contacts[]` (ชื่อ, โทร, ตำแหน่ง)
- `lineUserId?` (สะพานไป LINE Ops แบบไม่บังคับ)
- `status` (active | inactive | …)

**Employee (พนักงาน)**

- `id` (คงรูปแบบ `SKC…` ได้)
- ชื่อ, `phone?`, `position`, `status`
- ฟิลด์ล็อกอินเหมือนปัจจุบัน (username/password หรือเทียบเท่า)
- `lineUserId?` (สะพานไปชีต staff ทีหลังได้)

### 4.2 Spine (กระดูกงานขาย→ปิดเงิน)

**Deal (ดีล)** — `type` ตรงกับบริการ 3 อย่างที่บอท LINE ถาม

- `id`, `customerId`
- `type`: `bigcleaning` | `maid` | `office`  
  (Big Cleaning / แม่บ้านประจำ / Sangkan Office — ไม่รวมเป็น `recurring` ก้อนเดียว)
- `stage`: ค่าที่อนุญาตขึ้นกับ `type` (ดูตาราง stage ต่อ type ใน §11.1)  
  ค่าที่เป็นไปได้ในระบบ: `lead` | `survey` | `costed` | `quoted` | `approved` | `ops` | `delivered` | `invoiced` | `lost`  
  (ไม่ใช่ทุก type ใช้ครบทุกค่า)
- `title` / ชื่อโครงการ, มูลค่าดึงจากใบเสนอราคาเมื่อมี
- `packageCode?` สำหรับ `office` / `maid` เมื่อใช้แพ็ค (เช่น S|M|L)
- กระดาน CRM อ่าน/เขียน**เฉพาะ** `Deal.stage` (+ ฟิลด์แสดงผล) · **แท็บแยกตาม type** · โชว์เฉพาะคอลัมน์ที่ type นั้นใช้

**กฎเปลี่ยน stage:** มี `STAGE_GRAPH[type]` ในโค้ด — เลื่อนได้เฉพาะขั้นที่อนุญาตของ type นั้น (กันสเตจผิดชนิด)

**Survey (สำรวจ)** — `dealId`, ข้อมูลหน้างาน, รูป, ต้นทุน  

**Quotation (ใบเสนอราคา)** — `dealId`, `customerId`, รายการ, ยอดรวม, สถานะ (`draft` | `sent` | `approved` | `rejected`)  

**Job (งานหน้างาน)** — `dealId`, ตารางเวลา, ทีม/`employeeIds`, สถานะ (`scheduled` | `in_progress` | `done` | `cancelled`)  

**DeliveryNote (ใบส่งมอบ)** — `jobId`, หลักฐานก่อน/ระหว่าง/หลัง  

**Document (เอกสารบัญชี/ขาย)** — หลายชนิดในโครงเดียว:

- `kind`: `deposit` | `deposit_receipt` | `invoice` | `tax_invoice` | `receipt` | `cash_sale` | `credit_note` | `debit_note`  
  (รวมใบเสนอราคาในมุม “ส่งเอกสาร” ได้ หรือส่งจาก Quotation โดยใช้กลไกเดียวกัน)
- `dealId?`, `customerId`, ยอดเงิน, สถานะ, โทเคนแชร์สาธารณะ (`/share?...`)
- ประวัติการส่ง (optional): ช่องทาง, เวลา, สำเร็จ/ล้มเหลว

**การส่งเอกสารให้ลูกค้า (มาตรฐาน 10/10 — ต้องมีก่อนเปิด)**

กด **ส่งเอกสาร** เปิดชีทเดียว ชัดเจน ไม่ซ่อนทางหลัก:

| ลำดับ | ช่องทาง | พฤติกรรม |
|-------|---------|----------|
| 1 (หลัก) | **คัดลอกลิงก์** `/share` | ปุ่มใหญ่ · ใช้ได้เสมอ · กดแล้ว toast “คัดลอกแล้ว” |
| 2 (รอง) | **ส่งเข้า LINE ลูกค้า** | เปิดใช้เมื่อมี `lineUserId` · ปุ่มชัด · ยืนยันก่อนส่งถ้าเพิ่งส่งไปภายใน 10 นาที |
| — | อีเมล | เฟสถัดไปหลังเปิด ไม่กั้นวันเปิด |

กฎ:

1. ไม่มี `lineUserId` → ปุ่ม LINE เป็นสถานะรองพร้อมข้อความสั้น “ยังไม่ผูก LINE” + ลิงก์ **ผูก LINE ด่วน** (ช่องวาง/สแกน userId) ในชีทเดียวกัน — **ไม่บล็อกคัดลอกลิงก์**  
2. แยกจาก notify กลุ่มบัญชี (`LINE_ACCOUNTING_GROUP_ID`) คนละปลายทาง คนละปุ่ม  
3. Messaging API `push` ไปที่ `customer.lineUserId` พร้อมลิงก์ `/share`  
4. บันทึกประวัติส่งทุกครั้ง (ช่องทาง, เวลา, ok/fail) แสดงใต้ปุ่มส่ง  
5. รองรับจาก QT, วางบิล, ใบกำกับ, ใบเสร็จ เป็นอย่างน้อย (ชนิดอื่นตามที่มีในเมนู)

**Payment (รับชำระ)** — `documentId`, ยอด, วิธีชำระ, วันเวลา, สถานะ  

**Expense (ค่าใช้จ่าย)** — ยอด, หมวด, ผูก `dealId` ได้ (ไม่บังคับ)

### 4.3 กฎเหล็ก (Invariants)

1. เอกสารขายต้องมี `customerId` เสมอ
2. Quotation / Job / Survey ต้องมี `dealId`
3. ขั้น CRM เก็บที่ Deal เท่านั้น (ไม่ใช้ JSON ใน settings เป็นแหล่งจริงคู่ขนาน)
4. กำไรต่อดีล = เงินรับ (หรือเอกสารรายได้ที่ออกแล้ว) − (ต้นทุนจริงของ Job + ค่าใช้จ่ายของดีล)

```text
Customer ──┬── Deal ── Survey
           │      ├── Quotation
           │      ├── Job ── DeliveryNote
           │      ├── Document ── Payment
           │      └── Expense
Employee ──┴── ถูก assign ใน Job
```

## 5.1 หน้า Topology Graph — รวมฟังก์ชันภายในระบบ

Topology ไม่ใช่แค่แผนที่ข้าม repo แต่เป็น**แผนที่ฟังก์ชันของ ERP** เป็นหลัก แล้วค่อยโยงออกนอก

### มุมมอง 2 ชั้น (สลับได้)

| มุมมอง | โชว์อะไร |
|--------|----------|
| **ภายใน (ค่าเริ่มต้น)** | โมดูล/ฟังก์ชันใน `sangkan-clean` และเส้นข้อมูลระหว่างกัน |
| **ทั้งบ้าน** | เพิ่มโหนดเว็บ / บอท / LINE Ops เป็นบริบทภายนอก |

### โหนดฟังก์ชันภายในที่ต้องมี (อย่างน้อย)

| กลุ่ม | ฟังก์ชัน (โหนด) |
|------|------------------|
| คน | ลงทะเบียนลูกค้า · ลงทะเบียนพนักงาน · บัตร/QR login · โปรไฟล์บริษัท |
| ขาย | CRM (3 ประเภทบริการ) · สำรวจ · ใบเสนอราคา · ส่งเอกสาร (ลิงก์ + LINE ลูกค้า) |
| เอกสารบัญชี | มัดจำ · วางบิล · ใบกำกับ · ใบเสร็จ · ขายสด · ลดหนี้/เพิ่มหนี้ · แชร์ `/share` |
| งาน | จองคิวทีม · ปฏิบัติการ · ใบส่งมอบ · รายงานต้นทุนจริง |
| คลัง | สต็อก · ใบเบิก · อนุมัติ · PO · คุมอุปกรณ์ |
| การเงิน | ค่าใช้จ่าย · วางบิลหลังปิดงาน · กำไรต่อดีล |
| เชื่อมต่อ | แจ้งกลุ่มบัญชี LINE · ส่งเอกสารเข้า LINE ลูกค้า · (ภายนอก) เว็บ/บอท |

### พฤติกรรมหน้าจอ

1. คลิกโหนดฟังก์ชัน → แผงขวา: คำอธิบายสั้น + ลิงก์ไปหน้าจอนั้นในแอป + entity ที่แตะ  
2. ฟิลเตอร์ตามบริการ: Big Cleaning / แม่บ้าน / Office (ไฮไลต์เฉพาะเส้นที่เกี่ยวกับ type นั้น)  
3. แยกสี/ลายเส้น: เรียกภายในแอป · ส่งออก LINE · เส้นที่ยังเป็นงานคน (ยังไม่ออโต้)  
4. ข้อมูลกราฟมาจาก **registry ฟังก์ชัน** ในโค้ด (เช่น `topology-registry.ts`) ไม่ hardcode มั่วใน UI — พอเพิ่มโมดูลใหม่ให้อัปเดต registry

### ไม่ใช่หน้าที่ของ Topology

- ไม่ใช่แดชบอร์ดตัวเลขขาย (มีหน้า Home อยู่แล้ว)  
- ไม่แก้ข้อมูลธุรกิจบนกราฟ (รอบแรกอ่านอย่างเดียว)  
- ไม่แทนที่ CRM board

## 6. แนวทางเทคนิค

### 6.1 การเก็บข้อมูล

เปลี่ยนจาก “ทุกอย่างเป็น DataRecord ชีตชื่อหลวม” ให้เป็น**โมเดลหลัก**แบบ Prisma ชัดเจนสำหรับเอนทิตีแกน

- มีชั้นอ่านของเก่าระยะสั้นได้ ถ้าต้องย้ายข้อมูลจาก `DataRecord`
- สถานะปลายทาง: ตารางเอนทิตีแกนเป็นของจริง · JSON ใน settings เหลือแค่คอนฟิกจริงๆ (`companyProfile`, อัตราต้นทุน)

### 6.2 API

- ตัวจัดการ CRUD ของ master + spine แบบมีชนิดข้อมูล
- ตัวตรวจร่วม (zod หรือเทียบเท่า) สำหรับเปลี่ยน stage และสร้างเอกสาร
- คงระบบล็อกอินคุกกี้เดิม เว้นแต่จะอนุมัติออกแบบ auth ใหม่แยกต่างหาก

### 6.3 การย้ายสู่ TypeScript (เจตนาจัดใหญ่ก่อนเปิด)

เพราะเลือกตัวเลือก **2** (จัดแกนใหญ่ให้จบก่อนออนไลน์):

1. ตั้งค่า TS + ความเข้มที่เหมาะกับโค้ดใหม่
2. สร้างโมเดลแกน + API เป็น TypeScript ก่อน
3. หัน CRM / ลูกค้า / สร้างใบเสนอราคา / จองคิวงาน มาเรียก API ใหม่
4. ย้ายหน้าอื่นตามจำเป็นจนผ่านเช็คลิสต์ก่อนเปิด
5. **ไม่บังคับ**แปลงไฟล์เก่าทุกไฟล์ ถ้าเรียก API ใหม่ถูกต้องแล้ว — แต่โค้ดแกน**ใหม่ทั้งหมด**ต้องเป็น TS

### 6.4 คุมความเสี่ยงตอนขึ้นออนไลน์ (แม้เลือกแบบ 2)

กังวลเรื่องตัดสลับยากมีเหตุผล ใช้เกราะดังนี้:

| เกราะ | วิธีทำ |
|-------|--------|
| ลองบน Staging ก่อน | ซ้อม happy path บน DB/VPS ทดสอบก่อนชี้โดเมนจริง |
| สำรอง DB | สแนปช็อต SQLite/วอลุ่มก่อนตัดสลับ |
| สคริปต์ย้ายข้อมูล | นำเข้าชีตเก่า → ID แกนใหม่ ทิศทางเดียว · dry-run + รายงานรายการกำพร้า |
| Feature flag | ช่วง fallback อ่านอย่างเดียวได้เป็นวัน ไม่ใช่เป็นสัปดาห์ |

## 11. มาตรฐาน 10/10 (ถูกต้อง · สะดวก · ง่าย) — บังคับ

เป้าหมายคะแนนหลังทำตามสเปกนี้: **10 / 10 / 10**  
ข้อใดขาด = ยังไม่ถือว่าพร้อมอนุมัติเปิดจริง

### 11.1 ความถูกต้อง — stage ต่อประเภทบริการ

| type | คอลัมน์ CRM ที่โชว์ | stage เริ่มต้น |
|------|---------------------|----------------|
| `bigcleaning` | สำรวจ → ประเมิน → เสนอราคา → อนุมัติ → งาน → ส่งมอบ → วางบิล | `survey` |
| `maid` | ดีลใหม่ → เสนอราคา/แพ็ค → อนุมัติ → รอบงาน → เอกสาร | `lead` |
| `office` | ดีลใหม่ → แพ็ค S/M/L → เอกสาร | `lead` |

- คอลัมน์ที่ไม่มีในแถว = **ไม่เรนเดอร์**  
- API ปฏิเสธการย้าย stage นอกกราฟของ type นั้น  
- `lost` มีได้ทุก type จากเมนูบนการ์ด

### 11.2 ความสะดวก — ทางลัดบังคับมี

| จุด | ต้องมี |
|-----|--------|
| `/customers` | ปุ่มสร้างดีล → เลือก bigcleaning / maid / office |
| ฟอร์มดีล / QT | modal สร้างลูกค้าด่วน (ชื่อ+โทรอย่างน้อย · taxId ทีหลังได้) |
| การ์ดดีลใน CRM | ปุ่มลัด: เปิด QT · ส่งเอกสาร · เปิดงาน (ตาม stage) |
| เอกสาร | ชีทส่งเอกสารตาม §4 (ลิงก์หลัก · LINE รอง) |
| ผูก LINE | ใส่/วาง `lineUserId` ได้จากชีทส่งเอกสารหรือหน้าลูกค้าโดยไม่ต้องออกจาก flow |
